## 二、需求文档怎么写（四份文档的结构模板）

### 文档一：需求分析文档

```markdown
# 会务管理系统 — 需求分析文档

## 1. 系统概述
- 背景：为各类会议提供线上管理平台
- 目标用户：参会者、会议组织者、系统管理员
- 核心价值：一站式完成会议创建、报名、缴费、签到、入住管理

## 2. 角色分析
| 角色 | 权限描述 |
|------|---------|
| 参会者 | 注册登录 → 浏览/加入会议 → 填写参会信息 → 缴费 |
| 组织者 | 注册登录 → 创建/管理会议 → 管理特邀码 → 签到入住管理 |
| 管理员 | 登录 → 审核会议 → 管理用户 → 系统维护 |

## 3. 功能需求（按模块）
### 3.1 用户系统（M1）
- 注册：用户名、密码、手机、邮箱，角色选择（参会者/组织者）
- 登录：身份验证 + 角色区分
- 个人中心：查看/修改个人信息
- 权限拦截：未登录跳转登录页，角色不匹配拦截

### 3.2 会议管理（M2）
- 创建会议：标题、描述、地点、起止时间
- 编辑/删除会议（仅创建者）
- 特邀码管理：生成、查看使用情况、设置最大使用次数
- 我的会议列表

### 3.3 参会管理（M3）
- 浏览所有已审核通过的会议
- 按关键词搜索会议
- 通过邀请码加入会议
- 填写参会信息：往返时间、住宿要求、其他情况

### 3.4 缴费管理（M4）
- 虚拟缴费：选择参会记录 → 支付（模拟）
- 缴费记录查询
- 缴费状态查看（已缴费/未缴费）

### 3.5 签到入住管理（M5）
- 参会者签到（手动签到/按姓名搜索）
- 签到统计（已签到人数/未签到人数）
- 房间分配与入住状态管理

### 3.6 管理员面板（M6）
- 会议审核：查看待审核会议 → 通过/驳回
- 用户管理：查看、禁用/启用用户
- 系统信息维护

## 4. 非功能需求
- 页面统一使用 Bootstrap 5 风格
- 所有操作有反馈提示（成功/失败）
- 密码加密存储（MD5 + 盐值）
- 同一账号不能同时在多地登录（可选）
```

## 三、分工调整建议

原方案基本合理，但有**两个问题**需要调整：

### 问题1：M3（参会管理）比 M4（缴费管理）重很多

M3 包含浏览、搜索、加入会议、填参会信息，工作量明显大；M4 只有缴费 + 记录查询。

**建议调整**：把"填写参会信息（往返时间、住宿要求）"从 M3 挪给 M4，让 M4 变成"参会信息 + 缴费"合并模块。这样每人工作量更均衡：

| 成员 | 调整后模块 |
|------|----------|
| M3 | 浏览会议列表、搜索会议、邀请码加入会议 |
| M4 | 填写参会信息（往返/住宿/其他）+ 虚拟缴费 |

### 问题2：M5（签到入住）开发时需要 M3 的参会者数据

M5 做签到入住时，必须基于已有的参会记录（attendees 表）。**组长需要先建好 attendees 表 + DAO**，或者 M3 先提供测试数据。

### 最终的分工方式：每人提交自己模块的需求

| 步骤 | 谁做 | 产出 |
|------|------|------|
| 1. 数据库设计定稿 | 组长 + 全员 review | `03-数据库设计文档.md` |
| 2. 接口定义 | **每人写自己模块的接口** | 汇总到 `04-接口设计文档.md` |
| 3. 页面原型 | **每人画自己模块的 JSP 草图** | 手绘/截图都行 |
| 4. M1 搭建项目框架 | 组长先做 | Maven 项目结构 + 公共代码 |
| 5. 编码 | 每人按分支开发 | 按 M1→M2+M3→M4+M5+M6 顺序 |
| 6. 组长创建 SQL 脚本 | 组长 | `sql/init.sql` + 测试数据 |
| 7. 集成联调 | 全员 | 修复接口不匹配问题 |
| 8. 部署上线 | 组长 | `http://47.113.22.1/` |

## 四、操作建议（本周即可开始）

1. **现在**：组长建 GitHub 仓库，把上面四份文档模板 push 到 `docs/` 目录
2. **2天内**：每人写自己模块的接口设计（按文档四的表格格式），交给组长汇总
3. **3天内**：解决 MySQL 密码问题 + 执行建表语句
4. **4天内**：组长搭建 Maven 骨架项目（目录结构 + pom.xml + 公共 Filter/工具类），push 到 main 分支
5. **第5天起**：各成员从 main 拉取，在自己分支上开始编码
