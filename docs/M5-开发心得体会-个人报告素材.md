# M5 签到与入住管理模块 — 开发心得体会

> 模块：会务管理系统 — M5 签到与入住管理
> 角色：模块负责人（成员D）
> 开发周期：2026-05-30 ~ 2026-06-01
> 技术栈：Jakarta EE 10 + Servlet + JDBC + MySQL 8.0 + JSP + Bootstrap 5 + AJAX

---

## 一、模块概述

我负责的 M5 模块是会务管理系统中组织者角色的核心功能，包含两个子模块：

| 子模块 | 功能 | 涉及数据表 |
|--------|------|-----------|
| 签到管理 | 参会者签到、签到列表查询、签到统计 | `checkins`, `attendees`, `users` |
| 入住管理 | 房间分配、入住状态管理、房间列表查询 | `accommodations`, `checkins`, `attendees` |

完整开发链路：**VO → DAO 接口 → DAO 实现 → Service → Servlet → JSP 页面 → SPA 集成**，共涉及 3 个 VO、4 个 DAO、2 个 Service、1 个 Servlet、2 个 JSP 页面。

## 二、开发过程与遇到的问题

### 2.1 环境与依赖配置问题

开发初期最棘手的问题不是业务逻辑，而是**环境配置**。当我拉取项目代码后，`mvn compile` 直接失败，一连串的问题浮现：

**JDBC 驱动缺失**：项目的 `pom.xml` 初始没有 MySQL JDBC 驱动依赖。我在 `JDBCUtil.java` 中看到 `Class.forName("com.mysql.cj.jdbc.Driver")`，但 Maven 依赖树中找不到对应的 JAR。排查后发现需要手动添加 `mysql-connector-j` 依赖（注意不是旧的 `mysql-connector-java`，Jakarta EE 10 时代 artifactId 已变更）。

**JDBC 连接串参数不全**：加上驱动后，连接 MySQL 8.0 时报 `Public Key Retrieval is not allowed`。这个错误信息并不直观，查阅 MySQL 8.0 文档后才知道，新版本默认禁用公钥检索，需要在 URL 中显式开启 `allowPublicKeyRetrieval=true`，同时还要处理时区问题（`serverTimezone=Asia/Shanghai`）避免 `LocalDateTime` 出现 8 小时偏差。

**JSTL 标签无法解析**：JSP 页面中使用 `<c:if>` 时报 `The absolute uri cannot be resolved`。问题出在两个层面：一是缺少 JSTL 依赖，二是 Tomcat 10 使用 Jakarta EE（包名 `jakarta.*`）而非旧的 Java EE（`javax.*`），必须用 `jakarta.servlet.jsp.jstl` 而非 `javax.servlet.jsp.jstl`。添加依赖后仍未解决，最后发现是 `target/` 目录残留了旧版本 JAR，`mvn clean` 后问题消失。

这段经历让我深刻认识到：**环境问题往往比业务代码更消耗时间**，团队统一开发环境、写好 `pom.xml` 是项目启动阶段最重要的事情。

### 2.2 数据层开发中的隐蔽陷阱

#### Map.of() 的空指针异常

在 `CheckinDaoImpl.getCheckinList()` 中，我最初使用 Java 9 的便捷工厂方法构造返回数据：

```java
Map<String, Object> row = Map.of(
    "attendeeId", rs.getInt("attendee_id"),
    "username", rs.getString("username"),
    "phone", rs.getString("phone")  // ← 这里！
);
```

代码编译通过，本地测试也正常。但部署到 Tomcat 后，某次查询包含手机号为 NULL 的参会者时，直接抛出 `NullPointerException`，堆栈指向 `Map.of()`。我一开始完全没想到是 `Map.of()` 的问题，因为报错信息显示是数据库查询结果处理异常，我反复检查 SQL 和 ResultSet 读取逻辑都没有问题。

最后在查阅 Java 文档时发现：**`Map.of()` 和 `List.of()` 明确禁止 null 键和 null 值**。而数据库字段可能为 NULL（如 `phone`），当 `rs.getString("phone")` 返回 null 时，`Map.of()` 立即抛 NPE。

**解决思路**：全面放弃 `Map.of()` / `List.of()`，改回 `HashMap` / `ArrayList`。虽然代码行数略增，但 null 安全对于数据库查询场景至关重要。这个教训让我养成了一个习惯：涉及外部数据源（数据库、API响应）时，永远假设数据可能为 null，不使用禁止 null 的不可变容器。

#### Jackson 序列化 Java 8 时间类型

在返回签到列表的 JSON 中，`LocalDate` 字段被序列化为 `[2026,5,30]`（数组格式），而不是预期的 `"2026-05-30"` 字符串。前端 JavaScript 无法正确解析这种格式。

问题根源是 Jackson 默认使用 Java 的 `Date`/`Calendar` API，对 Java 8 引入的 `java.time` 包（JSR-310）没有内置支持。解决方法是注册 `JavaTimeModule`：

```java
private ObjectMapper mapper = new ObjectMapper()
    .registerModule(new JavaTimeModule());
```

同时需要在 `pom.xml` 中确保有 `jackson-datatype-jsr310` 依赖。这个问题暴露了我对 Jackson 序列化机制理解不够深入——**第三方库的默认行为需要主动配置，不能想当然**。

#### 数据库设计文档与实际表结构不一致

按照设计文档，`checkins.checked_by` 的外键应该指向 `users.id`。但我在写 Service 层的签到校验逻辑时，发现实际数据库的外键指向的是 `conferences.organizer_id`。这意味着 `checked_by` 字段存的不是任意操作者的 ID，而是会议组织者的 ID——只有组织者才能执行签到。

如果我没有检查实际的建表 SQL 而完全信任设计文档，写入的 `checked_by` 值会因为外键约束失败。这个教训让我学会了：**开发前必须对照实际数据库结构（`SHOW CREATE TABLE` 或建表 SQL 脚本），不能只看设计文档**。

### 2.3 前后端集成中的问题

#### 非法日期格式导致 HTTP 500

在入住管理的房间分配功能中，前端使用 `<input type="date">` 选择入住和退房日期。正常情况浏览器会发送 `yyyy-MM-dd` 格式的字符串。但在测试时我发现，如果用户清空日期输入框、或使用不兼容的浏览器，后端 `LocalDate.parse()` 会直接抛出 `DateTimeParseException`，Servlet 层面因为没有捕获这个异常，最终返回 HTTP 500 给用户。

**排查过程**：
1. 先在浏览器 DevTools Network 标签中看到返回了 500 状态码
2. 查看 Tomcat 控制台，看到 `DateTimeParseException` 的完整堆栈
3. 定位到 `AccommodationService.assignRoom()` 中的 `LocalDate.parse(checkinDate)`
4. 确认异常没有被任何 try-catch 包裹

**解决方案**：在 Servlet 的 `handleAssignRoom()` 方法中添加显式的异常捕获：
```java
} catch (java.time.format.DateTimeParseException e) {
    sendError(resp, 400, "日期格式错误，正确格式为 yyyy-MM-dd");
    return;
}
```

这个修复虽然简单（只加了 4 行代码），但反映了一个重要原则：**Controller 层不能信任任何来自客户端的输入**。无论是 URL 参数、表单数据还是 AJAX body，都应该做防御性校验，异常必须有明确的错误响应而不是让框架返回 500。

#### SPA 框架集成中的两个细节

项目的用户主界面 `index2.jsp` 采用 SPA 架构：侧边栏菜单点击 → AJAX 加载 JSP 片段 → `innerHTML` 插入内容区。我的两个 JSP 页面需要作为内容片段嵌入其中。

遇到的第一个问题是 **`innerHTML` 不执行 `<script>` 标签**。我的 JSP 页面中包含了大量 JavaScript 逻辑（加载会议列表、处理签到按钮、SweetAlert2 弹窗等），通过 `innerHTML` 插入后脚本完全不运行。排查后发现在 `index2.js` 的 `loadPage()` 函数中已经处理了这个问题——手动遍历 `<script>` 标签并用 `document.createElement('script')` 重建。这个知识点我之前并不了解，是通过阅读组长写的框架代码才学到的。

第二个问题是 **AJAX 加载的页面片段不能包含完整 HTML 结构**。我的 JSP 最初写了 `<html>`、`<head>`、`<body>` 等完整标签，嵌入后导致页面出现嵌套的 HTML 结构。解决方法是让 JSP 只包含 `<div class="content-wrapper">...</div>` + `<script>...</script>` 片段，作为"内容组件"而非"完整页面"。

### 2.4 跨模块协作问题

M5 模块强依赖 M2（会议管理）和 M3（参会管理）的数据。开发过程中，我的 JSP 页面需要通过 AJAX 调用 `/conference/myList` 获取组织者的会议列表来填充下拉框。但 M2 的组员当时只实现了会议搜索功能，没有提供这个接口。

按照常规流程，我应该等 M2 完成后再联调。但这会严重拖慢我的进度。我和组长沟通后决定**主动补齐依赖**：在 `ConferenceServlet` 中添加了 `/conference/myList` 端点，实现了 `ConferenceDaoImpl.findByOrganizerId()` 方法。这样 M5 可以独立开发和测试，不阻塞于其他模块的进度。

这个经历让我认识到：**等待是被动的做法，主动补齐缺口是更高效的选择**。在团队协作中，遇到阻塞时与其等待，不如先让功能跑通，后续再协商接口的归属和规范。

## 三、解决问题的通用思路

回顾整个开发过程，我总结出一套适用于后端开发的排查方法：

1. **看日志**：Tomcat 控制台输出是最直接的线索。任何 500 错误先在控制台找异常堆栈，定位到具体的行号和异常类型。

2. **逐层排除**：从外到内排查——先确认 HTTP 请求是否到达 Servlet（看日志），再确认 Service 调用是否成功（加打印），最后确认 DAO 的 SQL 是否正确执行（数据库日志或 try-catch 打印 SQLException）。

3. **小步验证**：每写完一个 DAO 方法，写一个简单的 `main` 方法测试，不要等到整个模块写完再一起测试。这样出问题时范围很小，容易定位。

4. **阅读源码和文档**：遇到 `Map.of()` NPE、Jackson 序列化等问题时，直接查看官方文档是最高效的。搜索引擎上的博客文章有时会过时，`Map.of(null)` 这种问题在官方 Javadoc 中写得非常清楚。

5. **查看实际数据**：数据库里到底存了什么？用 `SELECT * FROM checkins` 直接看，不要靠猜测。开发过程中我多次用 Navicat 直接查看数据表确认状态。

## 四、测试与验证

M5 模块的测试分为几个层次：

**DAO 层测试**：先在数据库客户端中手动执行 SQL 语句，确认 SQL 语法和逻辑正确，再写入 Java 代码。例如签到列表的 LEFT JOIN 查询、签到统计的 COUNT 聚合，都是先在 Navicat 中验证过 SQL 结果才在 `CheckinDaoImpl` 中实现。

**Service 层测试**：在 Service 方法中临时添加 `System.out.println` 输出中间结果，确认业务逻辑正确。例如签到防重复校验（先查后插）、分配房间前必须已签到的前置条件检查。

**接口层测试**：用浏览器 DevTools 的 Network 面板观察 AJAX 请求和响应。特别注意：
- 返回的 JSON 结构是否与前端预期一致（字段名大小写要匹配）
- HTTP 状态码是否正确（200/400/401/500）
- 中文是否乱码（Servlet 的 `setContentType` 和 `setCharacterEncoding` 都要设置）

**尚未完成的测试**：
- 重复签到场景（attendee_id 有 UNIQUE 约束，应该返回友好提示而非 SQL 异常）
- 未签到就分配房间的前置校验
- 空参数请求的容错
- Session 过期后 AJAX 请求的处理

这些边界情况需要等 M3 模块完成后进行完整的联调测试。

## 五、收获与反思

### 技术层面

1. **深入理解了 Servlet + JDBC 的全链路开发**：从 HTTP 请求进入 Servlet，到 Service 层业务处理，到 DAO 层 SQL 执行，再到 JSON 序列化返回，完整走通了后端开发的每个环节。这个过程比课堂上的零散实验更有收获。

2. **掌握了容器化思维之外的"手动挡"开发**：课程要求不使用 Spring Boot、MyBatis 等框架，手写 SQL、手动管理连接、手动序列化 JSON。这让我理解了框架底层做了什么——比如 `@RestController` 自动做了 Jackson 序列化和异常处理，`JdbcTemplate` 自动管理了 try-with-resources。理解底层原理后，用框架时心里更有底。

3. **学会了排查配置类问题的方法**：JDBC 连接串参数、Maven 依赖冲突、JSTL 版本兼容性这些问题，错误信息往往不直观，需要对技术栈的版本和兼容性有一定了解。我学会了查看 Maven 依赖树（`mvn dependency:tree`）、查看 Tomcat 日志、检查实际加载的 JAR 版本等方法。

### 协作层面

4. **理解了模块间依赖管理的重要性**：M5 依赖 M2 和 M3，但每个模块的开发进度不同。与其被动等待，不如明确定义接口契约（URL、参数、返回值格式），双方按契约并行开发。如果对方来不及，可以自己先写一个最小可用版本（stub）让功能跑通。

5. **文档与代码同步的重要性**：数据库设计文档与实际建表 SQL 的不一致给我带来了困扰。以后在项目中，我会坚持"建表 SQL 是唯一真相源"的原则，设计文档只是辅助说明。

### 个人成长

6. **Debug 能力的提升**：这个模块的开发中我遇到的每个 bug 都需要自己排查和解决，从看日志到定位根因再到修复验证，形成了一套完整的 debug 流程。特别是在处理 `Map.of(null)` NPE 和 Jackson 序列化问题时的排查经验，让我对 Java 的运行时行为和第三方库的默认配置有了更深入的认识。

### 2.5 SPA 框架中的 DOMContentLoaded 陷阱

项目中 `index2.jsp` 采用 SPA 架构——侧边栏点击通过 AJAX 加载 JSP 片段到内容区。开发中期发现多个页面一直显示加载动画，数据请求根本没发出。排查了 API 接口、数据库连接都没有问题，最后定位到是 JavaScript 初始化逻辑的问题。

多个 JSP 页面（`conferenceHall.jsp`、`myConferences.jsp` 等）使用 `document.addEventListener('DOMContentLoaded', ...)` 来触发数据加载。在传统页面浏览中这完全正常，但通过 SPA 注入时，主页面的 `DOMContentLoaded` 早已触发，注入片段中新增的监听器永远不会被执行。这就解释了为什么组长那边页面正常、我们这边一直转圈——组长访问的是独立页面，而我们通过侧边栏 SPA 访问。

解决方案是将所有 SPA 片段的初始化从 `DOMContentLoaded` 监听改为直接调用函数。这个问题让我理解了单页应用中页面生命周期与传统多页面的本质区别。

### 2.6 数据库配置的跨环境冲突

每次从 main 分支合并代码，`JDBCUtil.java` 中的数据库密码都会被覆盖为其他人的配置（`123456`），导致本地 Tomcat 启动后所有数据库查询失败。这个问题在开发中反复出现——合并→密码变了→登录失败→排查→改回来。

根本原因是数据库连接配置硬编码在共享的 Java 文件中。理想的解决方案是将密码提取到 `.properties` 配置文件并加入 `.gitignore`，每个人维护自己的本地配置。但由于课程项目时间紧迫，这个改进没有实施，每次合并后手动修改密码成了惯例。

另一处数据库相关的问题是中文搜索失效。组长那边测试通过，我们这边却查不到任何中文关键词。排查发现 JDBC 连接串缺少 `characterEncoding=UTF-8` 参数，导致 `PreparedStatement.setString()` 传递中文参数时编码不一致。加上该参数后问题解决。这个排查过程让我意识到 MySQL 连接串中每个参数都有其必要性。

### 2.7 跨分支协作与 PR 管理

M5 模块的代码分布在多个文件中，其中不少是和其他模块共享的（如 `index2.jsp`、`ConferenceDao.java`）。在提交 PR 到 main 时，main 已经包含了 M6 管理员模块的新增方法，导致 `ConferenceDao.java` 等文件产生冲突。

解决这些冲突的策略是：M5 的方法全部保留，M6 新增的管理员方法也全部保留，双方互不干扰。同时顺手修复了一个合并代码中的 bug——`setUser_id()` 应为 `setUserid()`（VO 中的方法名不一致）。这次经历让我理解了频繁的小 PR 比一次性大 PR 更安全——我们 33 个文件的改动如果能分两批提交，风险会小很多。

### 2.8 团队协作中的设计讨论

加入会议的流程设计经历了多次迭代：最初所有用户通过邀请码加入 → 发现大家应该被标记为"特邀" → 增加 `join_source` 字段区分 → 讨论是否需要会议密码 → 最终简化为"直接加入"和"邀请码加入"双入口。

这个过程中我学到了功能设计不能闭门造车——需要结合需求文档、现有数据库结构、其他模块的开发进度综合考虑。提出的会议密码方案虽然合理，但涉及 M2、M3 的改动太多，需要和组员协商后调整。最终采用的双入口方案改动最小、效果直接，是更务实的选择。
