## 一、项目初始化阶段（2026-05-24 ~ 05-25）

### 1.1 .idea 文件夹反复提交到 Git

**时间**：2026-05-25  
**现象**：`.idea/` 目录（IDEA 项目配置）被反复提交到 Git 仓库，导致多人协作时产生大量冲突。

**排查过程**：
- 检查 `.gitignore` 文件，发现初始模板没有包含 `.idea/`
- 在 05-25 一天内有 **4 次** 提交都在处理 `.idea` 移除问题（commit `91a3b23`, `157ae52`, `59b62ed`, `4be3c83`）

**根因**：
1. `.gitignore` 创建太晚，`.idea/` 目录已经被 Git 追踪
2. 只加 `.gitignore` 不执行 `git rm --cached` 无法停止追踪

**解决方案**：
```bash
# 正确做法（两步）：
echo ".idea/" >> .gitignore
git rm -r --cached .idea/
git commit -m "chore: 移除 .idea 文件夹并更新 .gitignore"
```

**经验教训**：项目初始化第一步就应该创建 `.gitignore`，把 IDE 配置、编译产物、本地环境文件全部加入。

## 二、M1 用户系统（2026-05-25 ~ 05-30）

### 2.1 数据库连接配置

**时间**：2026-05-28  
**现象**：多个同学本地数据库名称不一致，导致 JDBC 连接失败。

**排查过程**：

**根因**：建库时没有统一命名约定。

**解决方案**：`JDBCUtil.java` 中数据库名统一为 `web_db`：
```java

```

**经验教训**：数据库名、表名、字段名必须在设计文档中明确，所有人严格按文档执行。

### 2.2 JDBC 连接参数问题

**时间**：2026-05-30  
**现象**：部分环境 JDBC 连接报错：
- `Public Key Retrieval is not allowed`
- 时区错误导致 `LocalDateTime` 转换异常

**解决方案**（commit `5575760`）：在 JDBC URL 中显式指定参数：
```java
private static String DB_URL =
    "jdbc:mysql://localhost:3306/web_db"
    + "?allowPublicKeyRetrieval=true"
    + "&useSSL=false"
    + "&serverTimezone=Asia/Shanghai";
```

| 参数 | 作用 |
|------|------|
| `allowPublicKeyRetrieval=true` | 允许客户端获取服务器公钥（MySQL 8.0+ 要求） |
| `useSSL=false` | 本地开发禁用 SSL |
| `serverTimezone=Asia/Shanghai` | 统一时区，避免 `LocalDateTime` 转换差 8 小时 |

### 2.4 JSTL 依赖与旧 JAR 冲突

**时间**：2026-05-30  
**现象**：JSP 页面使用 JSTL 标签（`<c:if>`, `<c:forEach>`）时报错 `The absolute uri: http://java.sun.com/jsp/jstl/core cannot be resolved`。

**排查过程**：
1. 检查 `pom.xml` — 缺少 JSTL 依赖
2. 添加依赖后仍然报错 — `target/` 目录中有旧版 JAR 残留
3. `mvn clean` 清理后重新编译通过

**解决方案**：
```xml
<!-- Jakarta JSTL（Tomcat 10 使用 Jakarta EE，不是 javax） -->
<dependency>
    <groupId>jakarta.servlet.jsp.jstl</groupId>
    <artifactId>jakarta.servlet.jsp.jstl-api</artifactId>
    <version>3.0.0</version>
</dependency>
<dependency>
    <groupId>org.glassfish.web</groupId>
    <artifactId>jakarta.servlet.jsp.jstl</artifactId>
    <version>3.0.1</version>
    <scope>runtime</scope>
</dependency>
```

**关键点**：Tomcat 10 使用 **Jakarta EE**（包名 `jakarta.*`），不是旧的 Java EE（`javax.*`）。如果用错包名或版本，JSTL 标签无法解析。

清理旧构建的命令：
```bash
mvn clean compile
```

### 2.6 UserServlet 接口整合

**时间**：2026-05-30

## 三、M2 会议管理与搜索（2026-05-26 ~ 05-30）

### 3.1 搜索功能的三级匹配策略

**时间**：2026-05-30  
**需求**：用户输入关键字搜索会议，需要容忍输入不完整/不精确的情况。

**实现方案**（commit `c87e6bf`）：三级降级搜索：

| 级别 | 策略 | SQL 示例 | 匹配效果 |
|:---:|------|---------|---------|
| 1 | 子串模糊匹配 | `LIKE '%全国大会%'` | "全国" → 匹配"全国大会" |
| 2 | 非连续字符匹配 | `LIKE '%全%国%大%会%'` | "全大" → 匹配"全国大会" |
| 3 | 单关键字匹配 | 拆字分别 `LIKE` | "全"或"国"或"大"或"会" |

**代码要点**：
- 第 2 级：`"%".join(keyword.split("")) + "%"` 构造非连续匹配模式
- 第 3 级：过滤非汉字/字母字符，逐个单字查询后合并去重

### 3.2 全文索引优化搜索性能

### 3.3 ConferenceServlet 接口按需补全

**时间**：2026-05-30  
**背景**：M5（签到入住）模块的 JSP 页面需要通过 AJAX 调用 `/conference/myList` 获取组织者的会议列表，但 M2 最初只实现了搜索功能，没有这个接口。

**解决方案**（commit `e145eba`）：在 `ConferenceServlet` 中添加 `/conference/myList` 端点：
```java
@WebServlet("/conference/*")
public class ConferenceServlet extends HttpServlet {
    @Override
    protected void doGet(...) {
        if ("/myList".equals(path)) {
            handleMyList(req, resp);  // 根据登录用户 ID 查询其创建的会议
        }
    }
}
```

**经验教训**：模块间依赖要提前沟通。M5 依赖 M2 的接口，如果 M2 不及时提供，M5 无法独立开发。**最佳实践：先定义接口契约（URL + 参数 + 返回值格式），双方按契约并行开发。**

## 四、M5 签到与入住管理（2026-05-30 ~ 05-31）

### 4.1 Map.of(null) 抛出 NullPointerException

**时间**：2026-05-30  
**现象**：运行时报 `NullPointerException`，堆栈指向 `Map.of()` 调用。

**根因**：`Map.of()` 和 `List.of()` 是 Java 9+ 的不可变集合工厂方法，**不允许 null 键或 null 值**。当参会者手机号在数据库中为 NULL 时，`Map.of("phone", row.getPhone())` 直接抛 NPE。

**错误代码**：
```java
// ❌ 如果 phone 是 null，Map.of() 抛 NPE
Map<String, Object> row = Map.of(
    "username", rs.getString("username"),
    "phone", rs.getString("phone")  // 数据库可能为 NULL
);
```

**修复**（commit `bfcf84b`）：全部改用 `HashMap`：
```java
// ✅ HashMap 允许 null 值
Map<String, Object> row = new HashMap<>();
row.put("username", rs.getString("username"));
row.put("phone", rs.getString("phone"));  // null 安全
```

**经验教训**：项目中凡是涉及数据库查询结果的地方，**禁止使用 `Map.of()` / `List.of()`**，因为数据库字段可能为 NULL。统一使用 `HashMap` / `ArrayList`。

### 4.2 Jackson 无法序列化 Java 8 时间类型

**时间**：2026-05-30  
**现象**：返回 JSON 时 `LocalDate` / `LocalDateTime` 字段序列化失败或格式异常：
- `LocalDate` 序列化为 `[2026,5,30]`（数组格式）而非 `"2026-05-30"`
- `LocalDateTime` 序列化异常

**根因**：Jackson 默认不支持 Java 8 时间 API（JSR-310），需要注册 `JavaTimeModule`。

**修复**（commit `bfcf84b`）：
```java
// ❌ 没有注册 JavaTimeModule
private ObjectMapper mapper = new ObjectMapper();

// ✅ 注册后 LocalDate/LocalDateTime 正常序列化为 ISO 字符串
private ObjectMapper mapper = new ObjectMapper()
    .registerModule(new JavaTimeModule());
```

**Maven 依赖**（`pom.xml` 中已有的依赖起了作用）：
```xml
<dependency>
    <groupId>com.fasterxml.jackson.datatype</groupId>
    <artifactId>jackson-datatype-jsr310</artifactId>
    <version>2.15.2</version>
</dependency>
```

### 4.3 非法日期格式导致 HTTP 500

**时间**：2026-05-31  
**现象**：用户在入住管理页面分配房间时，如果日期输入格式不正确（如空值、`"abc"`、非标准格式），服务端直接抛 `DateTimeParseException` 未捕获，返回 HTTP 500。

**排查过程**：
1. 前端用 `<input type="date">`，正常情况会传 `yyyy-MM-dd`
2. 但用户可能手动输入非法日期，或浏览器兼容性问题导致传空字符串
3. `LocalDate.parse()` 对非法输入直接抛异常，Servlet 未 catch

**修复**（commit `8e46f58`）：在 `CheckinServlet.handleAssignRoom()` 中添加显式捕获：
```java
try {
    String result = accommodationService.assignRoom(...);
    // ...
} catch (java.time.format.DateTimeParseException e) {
    sendError(resp, 400, "日期格式错误，正确格式为 yyyy-MM-dd");
    return;
}
```

**经验教训**：
- Controller 层要对所有用户输入做防御性校验
- `LocalDate.parse()` / `LocalDateTime.parse()` 要包在 try-catch 中
- 后端永远不要信任前端传过来的数据格式

### 4.4 M5 页面集成到 SPA 框架

**时间**：2026-05-31  
**背景**：项目使用 `index2.jsp` 作为主 SPA 框架（侧边栏 + AJAX 动态加载内容区），M5 的 `checkin_manage.jsp` 和 `room_manage.jsp` 需要作为内容片段集成进去。

**挑战**：
1. JSP 页面作为 AJAX 片段加载时，不能包含完整的 `<html>` / `<head>` / `<body>` 结构
2. 页面内的 JavaScript 需要通过 `contextPath` 变量获取应用路径
3. `innerHTML` 不会执行 `<script>` 标签，需要手动重新创建 script 元素

**解决方案**（commit `d902e02`）：
1. JSP 页面只包含 `<div class="content-wrapper">...</div>` + `<script>...</script>` 片段
2. `contextPath` 通过 `window.contextPath` 全局变量传递（在 `index2.jsp` 中用 JSP 表达式设置）
3. `loadPage()` 函数在设置 `innerHTML` 后手动执行内联脚本：
```javascript
var scripts = container.querySelectorAll('script');
scripts.forEach(function(oldScript) {
    var newScript = document.createElement('script');
    newScript.textContent = oldScript.textContent;
    oldScript.parentNode.replaceChild(newScript, oldScript);
});
```

**页面 URL 映射**（`index2.js` 中的 `getPageUrl()`）：
| menu data-page | 加载的 JSP |
|---------------|-----------|
| `checkin` | `/checkin_manage.jsp` |
| `room` | `/room_manage.jsp` |
| `default` | `/attendee/default.jsp` |
| `conferenceHall` | `/attendee/conferenceHall.jsp` |
| ... | ... |

### 4.5 数据库设计文档与实际表结构的差异

**发现时间**：2026-05-30  
**问题**：实际数据库表结构与设计文档存在以下差异，开发时必须注意：

| 差异点 | 设计文档 | 实际数据库 |
|--------|---------|-----------|
| `checkins.checked_by` 外键 | → `users.id` | → `conferences.organizer_id` |
| `accommodations.status` | 无此字段 | `TINYINT`（1=入住中/0=已退房） |
| `accommodations.checkin_date` | 可为 NULL | `NOT NULL` |
| `accommodations.checkout_date` | 可为 NULL | `NOT NULL` |

**影响**：
- `checked_by` 存的是 `organizer_id`，不是任意 user id——意味着只有组织者可以操作签到
- `accommodations` 插入时必须提供 `checkin_date` 和 `checkout_date`，否则 SQL 报错
- 页面显示入住状态时需要判断 `status` 字段

**经验教训**：数据库设计文档要与实际建表 SQL 保持同步。建议用 `sql/init.sql` 作为唯一真相源（single source of truth），设计文档标注"以 init.sql 为准"。

### 5.2 AJAX 请求 Session 过期处理

**问题**：AJAX 异步请求时，如果 Session 过期，后端返回 401，但前端不会像传统表单提交那样自动跳转登录页。

**解决方案**：在每个 AJAX 响应的 `.then()` 中统一检查 `code === 401`：
```javascript
.then(function(data) {
    if (data.code === 401) {
        location.href = ctx + '/login.jsp';
        return;
    }
    // ... 正常处理
});
```

---

### 5.3 fetch API 缓存问题

**问题**：`fetch('/conference/myList')` 在 Chrome 中可能被缓存，导致数据不刷新。

**解决方案**：添加 cache 控制和时间戳参数：
```javascript
fetch(ctx + '/conference/myList?_=' + Date.now(), { cache: 'no-store' })
```
